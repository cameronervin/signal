
# Context Engineering

Write new prompts or evaluate/improve existing ones using modern context engineering principles. The core insight: intelligence is not the bottleneck — **context is**. Your job is to curate what the model sees, when it sees it, and in what structure.

## Mode Detection

Determine which workflow to follow:

**Writing a new prompt?** → Follow the [Creation Workflow](#creation-workflow)
**Evaluating an existing prompt?** → Follow the [Evaluation Workflow](#evaluation-workflow)

## Creation Workflow

### 1. Gather Requirements

Before writing, understand:

- **Agent role**: What persona/expertise does the agent embody?
- **Task scope**: What specific task(s) must the agent perform?
- **Input shape**: What data enters the context window? (user messages, retrieved docs, state, tool results)
- **Output shape**: What structured output is expected? (JSON schema, tool calls, text)
- **Guardrails**: What must the agent never do? (security, scope limits, escalation rules)
- **Context lifecycle**: Single-turn or multi-turn? Long-running with compaction needs?

### 2. Draft the Prompt

Apply these principles in order:

**a) Keep core instructions under 300 words.** LLM reasoning degrades around ~3,000 tokens of prompt. The sweet spot for core instructions is 150–300 words.

**b) Structure with XML tags.** Separate role, tools, rules, output format, examples, and data:

```
<role>...</role>
<tools>...</tools>
<rules>...</rules>
<output_format>...</output_format>
<examples>...</examples>
```

**c) Position data correctly.** Longform data (documents, retrieved context) at the TOP. Query and instructions at the BOTTOM. This leverages attention patterns (models recall beginnings and endings best).

**d) Be explicit, not implicit.** State every constraint directly. Models follow literal instructions — they don't infer intent.

**e) Include 3–5 diverse examples** covering happy path, edge cases, and at least one negative example (what NOT to do).

**f) Define output format precisely.** Show the exact JSON schema, tool call format, or text structure expected.

**g) Add error recovery instructions.** Tell the agent how to handle failures, missing data, and ambiguity.

**h) Define termination criteria.** Prevent infinite loops — specify when to stop and how.

### 3. Validate Against Checklist

Run the [Prompt Quality Checklist](#prompt-quality-checklist) before finalizing.

## Evaluation Workflow

### 1. Read the Existing Prompt

Read the full prompt file. Also read the context injection mechanism (middleware, RAG pipeline, or message builder) that feeds data into the prompt.

### 2. Score Against Checklist

Run the [Prompt Quality Checklist](#prompt-quality-checklist). Track pass/fail for each item.

### 3. Identify Issues

Check for these anti-patterns:

| Anti-Pattern | Symptom | Fix |
|---|---|---|
| **Context dump** | Entire history/state loaded upfront | Select/compress — load only what's needed for the next decision |
| **Lost in the middle** | Critical instructions buried between large data blocks | Move instructions to top or bottom of context |
| **Implicit assumptions** | Prompt says "handle errors" without specifying how | Make every behavior explicit with examples |
| **Missing negative examples** | Agent over-triggers or handles out-of-scope requests | Add "do NOT" examples and boundary cases |
| **Fat prompt** | Core instructions exceed 300 words | Extract data/examples into separate context; compress instructions |
| **No output format** | Agent returns inconsistent shapes | Define exact schema with typed fields |
| **Framework-hidden prompt** | Prompt generated by framework, not inspectable | Own every token — hand-craft and version-control |
| **Stale context poison** | Old errors or resolved states persisting in context | Clear resolved errors; prune old tool results |
| **Monolith agent** | Single agent with 10+ tools and broad responsibilities | Decompose into focused specialist agents (3–8 tools each) |

### 4. Produce Report

Output findings using this format:

```markdown
## Context Engineering Review

**Prompt**: [file path]
**Score**: X/12 checklist items passing

### Findings

#### Critical (must fix)
- [Issue]: [Description] → [Specific fix]

#### Recommended (should fix)
- [Issue]: [Description] → [Specific fix]

#### Optional (nice to have)
- [Suggestion]

### Rewritten Prompt
[If requested, provide the improved version]
```

### 5. Offer to Implement

After presenting findings, ask the user whether to:
1. Implement all fixes
2. Implement critical fixes only
3. Provide the rewritten prompt without applying changes

## Prompt Quality Checklist

Score every prompt against these 12 criteria:

| # | Criterion | Pass Condition |
|---|---|---|
| 1 | **Concise core** | Instructions ≤ 300 words (excluding examples/data) |
| 2 | **XML structure** | Sections separated by XML tags (`<role>`, `<rules>`, `<output_format>`, etc.) |
| 3 | **Data positioning** | Longform data at top, query/instructions at bottom |
| 4 | **Diverse examples** | 3–5 examples including edge cases and negative examples |
| 5 | **Explicit output format** | Exact output shape defined (JSON schema, tool call, or template) |
| 6 | **No implicit assumptions** | All constraints stated directly; no reliance on model "common sense" |
| 7 | **Error recovery** | Instructions for handling failures, missing data, ambiguity |
| 8 | **Termination criteria** | Clear conditions for when the agent should stop or escalate |
| 9 | **Human escalation path** | Defined mechanism for deferring to humans on high-stakes decisions |
| 10 | **Context budget** | Just-in-time loading; no unnecessary history or state in the window |
| 11 | **Owned and versioned** | Prompt is hand-crafted, not framework-generated; stored in version control |
| 12 | **Eval coverage** | Positive and negative test cases exist or are specified |

## Context Window Management

When the prompt is part of a multi-turn or long-running agent, also verify:

| Strategy | When to Use | Implementation |
|---|---|---|
| **Write** | Long-running tasks | Persist state externally (scratchpad, memory files) |
| **Select** | Large knowledge bases | Retrieve only relevant context via RAG or just-in-time loading |
| **Compress** | Multi-turn conversations | Summarize and compact history; keep only last N tool results |
| **Isolate** | Multi-agent systems | Separate context windows per specialist agent |

Key rules:
- Prune old tool results (keep last 2–3)
- Clear resolved errors from context immediately
- Compact conversation history before context window fills
- Never dump raw state — serialize to minimal tokens
